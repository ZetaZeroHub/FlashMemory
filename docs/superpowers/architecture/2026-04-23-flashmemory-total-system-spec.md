# FlashMemory 总体系统规格 v0.1

- 日期：2026-04-23
- 状态：system spec draft
- 关系：承接 `2026-04-22-superpower-blueprint-design.md`，并与 `2026-04-23-memory-index-guide.md` 中的认知记忆主轴形成边界对照

## 1. 目标与边界

本规格定义 `FlashMemory` 作为项目主体的总体系统边界、模块归属、公开接口和演进路线。它不是对单一功能的补充说明，而是把 `FlashMemory` 的战略角色明确下来：

> `FlashMemory` 是对象世界、检索世界和路由世界的基础设施，不是记忆演化世界本身。

在这一定义下，`FlashMemory` 应承担以下职责：

- 统一摄取代码、文档、图片、PPT 等知识对象
- 将对象解析、分块、索引、图谱化
- 对用户意图进行低延迟路由
- 对 LLM 调用进行成本与质量路由
- 提供稳定的工具注册、调度与 MCP 暴露能力
- 向上游认知记忆系统提供可消费的对象、图谱、变化和路由信号

本规格明确不承担以下职责：

- 记忆的社会状态机
- 记忆晋升、降级、修复的策略闭环
- self-play harness 的自博弈评测层
- semiosphere、群体共识、传播漂移的认知治理
- 将 repo / file / commit 之类的代码对象写进认知内核

这些能力属于 `DeepMemory` 或其上层研究包，而不是 `FlashMemory` 的核心本体。

## 2. 总体分工判定

本项目蓝图中的五个子系统，按归属应当这样拆分：

| 蓝图子系统 | 归属 | 判定理由 |
| --- | --- | --- |
| 多模态文档解析器 | FlashMemory | 这是对象摄取与标准化入口，属于 substrate 层 |
| 意图路由引擎 | FlashMemory | 这是统一查询入口层，决定走哪条检索或处理管线 |
| LLM 智能路由器 | FlashMemory | 这是模型选择与成本控制，不是记忆意义本身 |
| 认知记忆架构 | DeepMemory | 这是解释、记忆、晋升、修复和演化的中轴 |
| Skills/Tool 工具路由平台 | FlashMemory 主导，DeepMemory 消费 | 工具注册与调度属于平台能力，但 DeepMemory 会基于其结果形成记忆 |

这个判定背后的原则很简单：

- 只要主要回答“是什么对象、在哪里、怎么搜、怎么路由”，就放在 `FlashMemory`
- 只要主要回答“这条主张该不该相信、该不该晋升、会不会漂移、怎么自我修正”，就放在 `DeepMemory`

## 3. 设计原则

### 3.1 对象优先，而不是文本优先

`FlashMemory` 的最小世界观不是文本块，而是对象图。

对象可以是：

- 代码符号
- 文档章节
- 论文概念
- 图片中的结构元素
- PPT 中的论点单元
- 工具定义与调用结果

文本只是对象的一种表面形式，不是系统的唯一真相。

### 3.2 低延迟路由优先于单模型万能

用户输入进入系统后，第一步不是直接调用大模型，而是做路由决策：

- 这是不是代码搜索
- 这是不是文档问答
- 这是不是图谱遍历
- 这是不是深度分析
- 这是不是工具调用

路由必须可解释、可观测、可回放。

### 3.3 索引、图谱、路由必须共享同一对象语义

解析、检索、路由、工具调度不能各自说各自的语言。它们必须共享同一套对象标识、概念归一与上下文定位方式。

### 3.4 工具平台必须是生态接口，不是硬编码插件表

工具能力应当通过统一注册、声明式描述和可观测调用来扩展，而不是把每个工具写死在主流程里。

### 3.5 与 DeepMemory 的交互必须通过 contract

`FlashMemory` 不能假设 `DeepMemory` 的内部记忆结构；`DeepMemory` 也不能依赖 `FlashMemory` 的内部实现细节。两者只通过稳定 contract 对接。

## 4. 系统分层

### Layer 1: Multimodal Ingestion

这一层负责把外部知识对象转成系统可处理的标准化输入。

职责包括：

- 文件类型识别
- PDF / PPT / 图片 / 文档解析
- 结构化块切分
- 概念节点抽取
- 解析质量标注

输出应至少包含：

- 标准化对象描述
- 结构化分块
- 锚点定位
- 基础 provenance

### Layer 2: Object Index and Graph Substrate

这一层负责把对象组织成可查询的知识底座。

职责包括：

- 统一索引代码块与文档块
- 建立对象节点与边
- 支持对象聚类与局部上下文聚合
- 支持概念节点与代码节点的语义连接

这一层是 `FlashMemory` 的“看见能力”。

### Layer 3: Intent Routing

这一层负责把用户请求分流到最合适的处理管线。

职责包括：

- 识别请求类型
- 在搜索、问答、图谱推理、工具执行之间选择路径
- 处理 fallback 与混合路由

这层应优先采用确定性或轻量语义路由，而不是在每次入口都调用大模型。

### Layer 4: LLM Routing

这一层决定调用哪个模型、给多少上下文、是否走本地或云端。

职责包括：

- 复杂度估计
- 模型等级选择
- 成本控制
- 上下文预算控制

这层可以参考 AST 复杂度、对象重要性、跨模块跨度、查询熵等信号。

### Layer 5: Tool and Skill Platform

这一层负责能力生态。

职责包括：

- 工具注册
- 工具发现
- 工具调用
- 工具声明
- MCP 暴露
- 社区能力扩展

### Layer 6: Observability and Control Plane

这一层负责让系统可观测、可治理、可回放。

职责包括：

- 日志
- 事件追踪
- 路由解释
- 调用审计
- 成本统计
- 失败回放

## 5. 模块规格

### 5.1 多模态文档解析模块

模块目标：

- 把非结构化知识纳入统一对象空间
- 让 PDF、PPT、图片、Markdown、代码片段共享可索引表示

建议输入：

- `path`
- `mime_type`
- `source_kind`
- `ingest_policy`

建议输出：

- `ParseArtifact`
- `Chunk`
- `ConceptNode`
- `Anchor`
- `ParseQuality`

关键约束：

- 解析结果必须保留原始定位
- 概念抽取不能覆盖原文 provenance
- 解析失败必须有降级输出，而不是静默吞掉

### 5.2 意图路由模块

模块目标：

- 在 10-100ms 量级完成入口分流

建议路由类别：

- `code_search`
- `doc_qa`
- `graph_reasoning`
- `deep_analysis`
- `tool_call`
- `fallback`

关键约束：

- 路由决策必须可解释
- 路由 utterance 集合可持续学习，但不能失去确定性边界

### 5.3 LLM 路由模块

模块目标：

- 将复杂问题分配给合适的模型

建议输入信号：

- 查询长度
- 实体密度
- 跨模块跨度
- 目标节点重要性
- 上下文预算

建议输出：

- `model_id`
- `context_budget`
- `reasoning_mode`
- `cost_hint`

关键约束：

- 不允许单一默认模型吞掉所有请求
- 低复杂度请求应优先本地模型

### 5.4 工具与技能平台模块

模块目标：

- 形成可扩展工具生态

建议能力：

- 统一注册
- 声明式工具定义
- 工具触发语句
- MCP 暴露
- 版本化工具能力

关键约束：

- 工具平台输出必须可回流到对象图
- 工具能力不能与路由逻辑硬耦合

### 5.5 知识图谱与对象检索模块

模块目标：

- 为上层系统提供稳定对象检索与关系遍历能力

建议能力：

- 代码对象检索
- 文档对象检索
- 概念关系遍历
- 结构化上下文回填

关键约束：

- 对象图谱必须与解析层共享统一锚点
- 检索结果必须能够回溯到原始对象与片段

## 6. 对 DeepMemory 的公开契约

`FlashMemory` 向 `DeepMemory` 提供的不是记忆解释，而是记忆原料。推荐公开能力包括：

- `ObjectResolution`
- `Search`
- `GraphTraversal`
- `ChangeSignal`
- `ProjectMetadata`
- `ToolInvocationRecord`

这些能力应保证：

- 稳定命名
- 稳定语义
- 稳定回溯
- 稳定版本化

`DeepMemory` 可以消费它们，但不能反向要求 `FlashMemory` 变成记忆内核。

## 7. 推荐目录与模块映射

建议 `FlashMemory` 的实现目录按能力组织，而不是按特定产品面组织：

```text
flashmemory/
  internal/
    ingest/
    parser/
    router/
    llm/
    graph/
    tool/
    telemetry/
    config/
  pip-package/
  docs/
  tests/
```

职责说明：

- `ingest/`：入口摄取与文件识别
- `parser/`：多模态解析与结构化切分
- `router/`：意图路由
- `llm/`：模型路由与调用封装
- `graph/`：对象图谱与检索
- `tool/`：技能注册与调度
- `telemetry/`：日志与审计

## 8. 端到端数据流

```text
用户输入
  ↓
意图路由
  ↓
  ├─ 代码 / 文档 / 图谱 / 工具 路由
  └─ LLM 路由
        ↓
  对象解析 / 图谱检索 / 模型推理 / 工具执行
        ↓
  结果回填对象图与事件日志
        ↓
  向 DeepMemory 暴露可消费的对象与变化信号
```

这条数据流的核心原则是：

- 先对象化，再解释化
- 先路由，再推理
- 先可回溯，再自动化

## 9. 与 DeepMemory 的边界

`FlashMemory` 不负责：

- 记忆状态机
- social_status 晋升
- interpretant 争议治理
- self-play harness
- semiosphere
- 传播漂移修复

`DeepMemory` 不负责：

- 文件解析
- 路由入口
- 工具注册中心
- 统一对象图谱底座

两者之间不是主从关系，而是 substrate 与 cognitive layer 的关系。

## 10. 实施阶段

### Phase 1: 统一对象底座

- 完成多模态解析入口
- 完成对象图谱统一锚点
- 完成最小意图路由

### Phase 2: 路由与工具平台

- 完成 LLM 路由
- 完成工具注册与 MCP 暴露
- 完成可观测与审计闭环

### Phase 3: 面向认知层对接

- 稳定公开 contract
- 为 DeepMemory 的 adapter 提供稳定 substrate
- 为群体记忆与演化层留出事件接口

## 11. 验收标准

`FlashMemory` 作为系统主体，至少应满足以下验收条件：

- 用户可以从任意知识对象进入统一摄取流程
- 用户可以在同一入口完成代码、文档、图谱、工具的路由
- 系统能够向上层提供稳定对象与变化信号
- 系统不会把记忆治理逻辑和 substrate 逻辑混在一起
- 系统不会因为追求通用性而丢失代码感知优势

## 12. 结论

本规格为 `FlashMemory` 划定了清晰边界：

- 它是认知基础设施的地基
- 它负责对象世界的统一摄取、索引、路由和工具调度
- 它通过 contract 为 `DeepMemory` 提供可演化的原料
- 它不承担记忆意义的治理本体

这意味着 `FlashMemory` 不是被削弱，而是被更准确地定位为：

> AI 认知基础设施的 substrate 层。
